# Workspace 指令（pz-skills 强路由配置）

> 这个文件是给 Claude Code 看的 **workspace 级 CLAUDE.md**。
>
> **使用方式**：把这份文件复制（重命名为 `CLAUDE.md`）到你想稳定使用 pz-skills 的某个目录里——比如一个专门的"觉醒/写作"工作目录。在该目录里启动 Claude Code（`cd 那个目录 && claude`），下面的强路由规则就会生效，让"需 workspace 加持"那一类 skill 在自然对话里也能稳定命中。
>
> 不在这个目录里启动 Claude Code 时，规则不生效——你的其他项目、日常对话完全不受影响。
>
> 任何时候都可以用 `/pz-skills:<skill-name>` 显式调用作为兜底，不依赖这份配置。

## Skill 强路由规则

下列规则的优先级**高于** Claude 的默认对话行为。当用户消息满足触发条件时，**必须**通过 Skill 工具调用对应 skill，不要先用默认对话模式响应。

### 触发 `pz-skills:kanjianzj`（看见自己 · 情绪与拖延陪伴）

用户消息中包含下面任意一类信号时调用：

- **情绪自陈**：「我焦虑/我难受/我崩溃/我心情很乱/我状态不对/累死了/烦死了/不想动/没劲/空落落」
- **自责自我否定**：「我没救了/我真没用/我又搞砸了/我又拖延了/又控制不住/我对自己很失望」
- **行为后悔**：「刷了一天手机/又对家人发火/又熬夜了/什么都没干/又没忍住」

→ 调用 `pz-skills:kanjianzj`，**不要先尝试共情或给建议**——让 skill 走"接住-命名-定位三脑-最小可行下一步"的流程。

### 触发 `pz-skills:jinzita`（金字塔 · 结构化表达）

用户消息中包含起草"给别人看的结构化文字"的请求：

- **写邮件/微信/汇报/提案**：「帮我写/起草/给... 写... 邮件/微信/汇报/提案/方案/总结」
- **写自媒体内容**：「帮我写小红书开头/公众号导语/PPT 大纲/视频脚本骨架」
- **改稿**：「这段话/这段文字 怎么改/帮我改/怎么说更清楚/让对方秒懂」
- **说服**：「我想跟 ... 说/我想说服 ... 做 ...」

→ 调用 `pz-skills:jinzita`，**不要直接吐成稿**——让 skill 先逼用户想清楚塔尖、做 MECE、设计 SCQA 开头。

### 例外（不要触发）

下列情况按默认行为处理，不要走上面的强路由：

- **用户已明确豁免**：「不用 skill / 直接给我写就好 / 别问那么多 / 我赶时间直接给答案」
- **场景天然不适合（jinzita）**：写小说、品牌 TVC、感情向朋友圈、头脑风暴/创意发散、亲密关系共情。
- **危机信号（kanjianzj 内部分支）**：用户提到自伤、自杀、伤害他人念头时，kanjianzj 自身会切到危机干预分支并给出专业求助资源——不要绕过。

## 维护说明（给 Ducc/Claude 自己看，未来加新 skill 时用）

> 凡是 `metadata.pz_skills_type: workspace_assisted` 的 skill，都需要在这个文件里加一段触发规则。`metadata.pz_skills_type: standalone` 的 skill（如 `xuexiqu`）不需要——它们 description 路由本身就够稳。
>
> 加新规则时按现有「触发 / → 调用 / 例外」的三段结构追加，保持文风一致。
>
> 每次 pz-skills 仓库新增 workspace_assisted 类 skill，必须**同步更新这份模板**，并在 commit message 里注明。详见 `MAINTENANCE.md`。
